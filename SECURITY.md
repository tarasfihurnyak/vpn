## Threat Model

The control plane is an internal admin API. The assumed threat actors and scenarios are:

| Threat | Description |
| --- | --- |
| Credential stuffing / brute-force | Automated attacks against the login endpoint |
| Stolen refresh token | Attacker replays a captured refresh token |
| CSRF | Malicious site triggers authenticated actions via browser cookies |
| Token forgery | Attacker fabricates or tampers with JWT access tokens |
| Privilege escalation | Regular user accesses admin operations |
| Data exposure via injection | SQL injection through user-supplied input |
| Man-in-the-middle | Traffic interception on the network |
| Sensitive data in logs | Passwords, tokens, or keys written to log output |

## Implemented Controls

### Transport Security

- **TLS termination** is handled by Traefik. The API container is never exposed directly on a public port.
- **HTTP → HTTPS redirect** is enforced at the Traefik entrypoint level. Plaintext connections are rejected.
- **Local development** uses certificates generated by [mkcert](https://github.com/FiloSottile/mkcert), which are signed by a locally trusted root CA. This avoids self-signed certificate warnings while keeping the setup realistic.
- The `certs/` directory is bind-mounted read-only into the Traefik container.

### Authentication

#### JWT Access Tokens

- Algorithm: **ES256** (ECDSA P-256). Asymmetric signing is used so that the public key can be distributed to future services for token verification without exposing the private key.
- The private key is loaded from the filesystem at startup and never leaves the server process.
- Access tokens are **short-lived** (default 15 minutes). Reducing lifetime limits the damage window of a stolen token.
- Tokens are validated on every protected request; expired or malformed tokens are rejected with `401`.

#### Refresh Tokens

- Refresh tokens are **opaque random values**. Only a SHA-256 hash is stored in the database; the raw token is never persisted.
- **Token rotation**: every `/refresh` call invalidates the old token and issues a new one.
- **Replay detection**: if a token that has already been revoked is presented, the server treats this as evidence of theft and **immediately revokes all active sessions for that user**.
- Refresh tokens are delivered and accepted exclusively via `HttpOnly` cookies to prevent JavaScript access.
- The `Secure` flag is set on cookies in production (`JWT_SECURE_COOKIE=true`), ensuring they are only sent over HTTPS.

#### Password Storage

- Passwords are hashed with **bcrypt** at the default cost factor (`bcrypt.DefaultCost = 10`).
- The raw password is never stored or logged.
- Authentication errors return a generic `invalid credentials` message regardless of whether the user exists, preventing username enumeration.

### Authorization

- All non-auth endpoints are protected by a JWT middleware that extracts and validates the access token from the `Authorization: Bearer` header.
- The authenticated user's UUID is injected into the request context; handlers use this to scope operations.
- **Role-based access control is not yet implemented** — see [Known Gaps](#known-gaps--planned-hardening).

### Input Validation

- **Request body size** is capped at 1 MB via middleware, preventing oversized-body denial-of-service attacks.
- **SQL injection** is not possible: all database queries are generated by sqlc and use parameterised queries via pgx. No raw string interpolation is used.
- UUIDs in URL parameters are validated by the Go standard library before reaching service logic.

### Brute-Force & Abuse Protection

- A **per-IP token-bucket rate limiter** is applied to `POST /api/auth/login`:
  - Sustained rate: 5 requests/minute
  - Burst allowance: 3 requests
- IP limiters are stored in memory and cleaned up after 10 minutes of inactivity to prevent unbounded memory growth.
- Exceeding the limit returns `429 Too Many Requests`.

### CSRF Protection

Cookie-based endpoints (`/api/auth/refresh`, `/api/auth/logout`) are protected by an **Origin header check**:

- The `Origin` header must be present and must match one of the configured allowed origins (`JWT_ALLOWED_ORIGINS`).
- Requests with a missing or disallowed origin are rejected with `403 Forbidden`.
- This is appropriate because cookie-driven endpoints are only reached by browsers, which always send the `Origin` header on cross-site requests. Non-browser clients (curl, CLI tools) use `Authorization: Bearer` and never reach these routes.

### Dependency & Build Security

- **Multi-stage Docker build**: the final image is a lean Alpine image containing only the compiled binary and CA certificates. No build toolchain, source code, or unnecessary packages are present.
- Dependencies are pinned via `go.sum`.
- `golangci-lint` is available via `make lint` to catch common issues before they reach CI. It also runs automatically on every push and pull request via GitHub Actions.

### Logging & Observability

- Structured JSON logging via zerolog.
- Sensitive values (passwords, token hashes, private keys) are never passed to the logger.
- CSRF and rate-limit rejections are logged at `WARN` level with the offending IP and path for audit purposes.

## Known Gaps & Planned Hardening

| Gap | Risk | Planned Fix |
| --- | --- | --- |
| No role-based access control | Any authenticated user can call any protected endpoint | Implement admin / user roles |
| No 2FA / MFA | Compromised credentials give full access | TOTP (2FA) and Google OIDC with domain restriction |
| No Swagger / API docs | Security review is harder without a formal spec | Generate OpenAPI spec |
| WireGuard server private key handling | Not yet designed; server-side key must be protected | Document key storage strategy; consider hardware-backed secrets |
| Secrets in `.env` file | `.env` can be accidentally committed or leaked | Use a secrets manager (Vault, AWS Secrets Manager) in production |
| No audit log | No record of who created/modified peers or users | Add an audit log table |
| No account lockout | Rate limiter resets after 10 min; persistent attackers can retry | Implement exponential backoff or temporary account lockout |
| DB connection not TLS-encrypted | Traffic between API and Postgres is plaintext (`sslmode=disable` by default) | Enable `sslmode=require` in production |
| No token revocation on password change | Existing tokens remain valid after a password reset | Invalidate all refresh tokens when password changes |
| Client WireGuard private key handling | Not yet designed; key must be generated and stored securely on the client machine and never transmitted | Document key storage strategy (filesystem permissions, OS keychain) |

## Secrets Management

The following secrets must be protected:

| Secret | Current Storage | Production Recommendation |
| --- | --- | --- |
| JWT ES256 private key | File on disk (`certs/jwt_private.pem`) | Mount from a secrets manager (Vault, AWS SSM); restrict file permissions to `0600` |
| WireGuard server public key | `server_config` table in PostgreSQL | Read-only access; safe to distribute to authenticated clients |
| Database password | `.env` file / Docker environment variable | Use a secrets manager or Docker secrets |
| mkcert root CA private key | Local machine only (not committed) | Not applicable in production; use a real CA |

Rules:
- The `certs/` directory and `.env` file **must** be listed in `.gitignore` and never committed.
- The JWT private key file should have permissions `0600` and be owned by the process user only.
- In production, prefer injecting secrets via environment variables at runtime rather than files on disk when possible.
